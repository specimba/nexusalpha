## 1. Project Landscape & Core Philosophy

### 1.1 Nexus OS A2A — The Orchestration Layer

Nexus OS is a **local‑first, project‑scoped Agent Operating System**. It treats agents as isolated processes, memory as a tiered biomimetic filesystem (S‑P‑E‑W), and the A2A protocol as a system bus. Over **6 sessions, 170+ automated tests, and 20+ git commits**, the project has survived catastrophic incidents (git deadlock, full sandbox wipe) and emerged with a hardened, resilient architecture.

**Core Philosophy:**
- **Governance‑first**: Deny‑by‑default isolation, immutable audit logs, non‑compensatory harm thresholds.
- **Resilience through mini‑services**: Long‑running processes survive container resets by running as PID 1 orphans.
- **Observe‑Gate‑Act pattern**: Every automation (cron, agent cycle) must first verify health, then gate on preconditions, then act.

### 1.2 DoppelGround — The Governed Research Spine

DoppelGround (also called ReviewGround) is a **trust‑first local research operating surface**. It transforms mixed research inputs (links, papers, repos, notes) into:

- Ranked and reviewable research artifacts,
- Dossier‑aligned mission suggestions,
- Readable session entry points (`OPEN_THIS_FIRST.md`, `SESSION_SUMMARY.md`),
- Bounded repo inspection notes (via Probe adapter).

**Key Differentiator:** DoppelGround **does not mutate the canonical research corpus** during isolated runs. It provides a governed evidence surface that can feed Nexus OS with cleaner task formation and better review visibility.

### 1.3 Collaboration Model

The two systems are complementary, not overlapping:

| **Layer** | **Responsibility** | **Owner** |
|---|---|---|
| Research Preparation | Ingest, rank, structure evidence; generate dossiers & missions | DoppelGround |
| Governance & Routing | Consume shaped research, decide swarm action, gate on confidence | Nexus OS |
| Execution & Identity | Assign agents, update identity cards, track performance | Nexus OS |

This clean division of labor prevents architectural collision and allows both systems to evolve independently while producing a joint **governance‑first autonomous swarm** capability.

## 2. Nexus OS Core Architecture

### 2.1 The Four Pillars (LOCKED)

| Pillar | Responsibility | MVP Implementation |
|---|---|---|
| **Bridge** | Inter‑agent communication | JSON‑RPC 2.0 + 4 Nexus headers (Project‑ID, Task‑ID, Lineage‑ID, Trace‑ID) |
| **Vault** | S‑P‑E‑W biomimetic memory | Session(RAM) → Episodic(FTS5) → Semantic(DB) → Wisdom(permanent) |
| **Engine** | DAG task orchestration | SHA‑256 task IDs, 30s heartbeat, state machine |
| **Governor** | Policy & safety | Deny‑by‑default, immutable JSON‑lines audit, KAIJU auth |

### 2.2 S‑P‑E‑W Memory Hierarchy

```
Session (RAM) → threshold → Episodic (FTS5) → extract → Semantic (DB→Zvec) → promote → Wisdom (permanent)
Wisdom → seed cold‑start → Session (bidirectional)
```

**Hybrid Retrieval Formula** (from MIA paper):
`score = 0.5 × semantic + 0.3 × value_reward + 0.2 × frequency_reward`

### 2.3 Security & Trust Model

- **SecretStore**: HMAC‑SHA256 with constant‑time comparison.
- **Hard‑fail encryption**: No silent plaintext fallback.
- **Trust Scoring**: Lane‑scoped Bayesian (Beta distribution) with time decay.
- **Scoring Events**: Append‑only audit trail with `hard_fail`, `finding_state`, and structured reason codes.
- **OWASP ASI Top 10** adopted as foundational threat model.

## 5. Information Architecture & Token Efficiency

### 5.1 The 5‑Layer Architecture

```
Layer 5: EXECUTION      → Do work (0 conversation tokens)
Layer 4: CHUNK PROTOCOL → Fit work in platform limits (~500 tokens)
Layer 3: PRESETS        → Fast recovery shortcuts (~800 tokens)
Layer 2: KNOWLEDGE      → Compressed reference files (~2,000 tokens, on‑demand)
Layer 1: GROUND TRUTH   → Single state file (~400 tokens)

TOTAL BOOT: ~3,700 tokens vs 50,000+ traditional = 93% savings
```

### 5.2 Compression Techniques

| Technique | Savings | How |
|---|---|---|
| Duplication elimination | ~60% | Remove repeated content, keep unique signals |
| Compression (tables > prose) | ~20% | Replace verbose explanations with tables |
| Index‑then‑read (presets) | ~10% | Presets point to files, don't inline everything |
| Reference‑not‑repeat | ~5% | Cross‑reference other agents' work |
| Boot sequence hierarchy | ~5% | Read only what you need, when you need it |

### 6.2 Agent Identification Cards (Proposed)

| Field | Purpose |
|---|---|
| Agent ID, Current Role | Identity |
| Domain Strengths, Failure Patterns | Capability profile |
| Evidence Discipline Score, Governance Compliance | Trust metrics |
| Confidence Calibration, Recommended Assignments | Routing guidance |

DoppelGround provides the **research‑side evidence** to populate these cards; Nexus OS maintains them across missions.

### 6.3 File‑Driven Coordination Pattern

```
[User Task] → [mem0 query] → [Hermes classification] → [Skill Registry match] → [Coordinator dispatch] → [.task.md creation] → [Worker execution] → [Result capture] → [Hermes + mem0 update]
```

Tasks are defined as YAML‑frontmatter Markdown files in `pending/`, `done/`, `failed/` queues. This enables any agent (GLM‑5, Kimi, MiMo) to pick up work without a central message broker.

## 8. Lessons Learned & Safeguards

### 8.1 Critical Lessons

| # | Lesson |
|---|--------|
| 1 | **Git is your only safety net.** Hourly auto‑commits are mandatory. |
| 2 | **NEVER use git checkout/branch for runtime state.** Use direct file writes. |
| 3 | **Design fallback chains for all dependencies.** mem0 v1.2 uses mem0ai → SQLite → in‑memory. |
| 4 | **Understand container lifecycle.** Mini‑services survive; shell sessions don't. |
| 5 | **Every automation needs a pre‑flight check.** Observe‑Gate‑Act prevents wasted cycles. |
| 6 | **Time budgets prevent runaway agents.** Cap every cron invocation. |

### 8.2 Safeguard Principles (Adopted)

1. **Boot Protocol**: Check dev server alive first. STOP if dead.
2. **Priority Ordering**: Bugs > Connections > Styling > Features.
3. **Time Budgets**: 8min/15min, 2min/1hr, 12min/6hr caps.
4. **Anti‑Pattern Bans**: NEVER git checkout/stash/nested crons.
5. **Graceful Degradation**: agent‑browser → curl fallback chain.
6. **Blocked Logging**: `[BLOCKED]` prefix for unfixable issues.
